Let's go for inventree documentation and see what do we need so we can integrate these functions inside it.

# Well after reading the documentation, the core mixins I will most likely use are BarcodeMixin, EventMixin, SettingsMixin, UserInterfaceMixin and maybe UrlsMixin and LocateMixin.
- BarcodeMixin : This one would be used to read the NFC tag, Inventree documentation explicitly frames bar code plugins as the extension point for custom scanned codes and supplier/product identification. In simple words reads a scanned NFC tag and resolve it to a drug / stock item.
- EventMixin : This one would be used for reacting to a stock item (or as it is said in the documentation "Part") when it's created or updated, this one would simplify the linking between a stock item and the NFC tag scanned. In simple words it auto-links a newly created stock item after the scan/write flow.
- SettingsMixin : This one would be used for storing the configuration and rules, maybe how the reader behave, or the mapping strategy, the tag formats...
- UserInterfaceMixin / UrlsMixin : This one would be used to add a custom server-side UI hooks or dedicated plugin endpoints inside InvenTree itself. In simple words add a custom UI widgets/pages inside InvenTree and expose endpoints for React.
- LocateMixin : Physically locate tagged stock in a custom way.

So the final app architecture would be:
- InvenTree backend plugin in Django : It uses the previous Mixin in its implementation.
- React frontend : This one would handle the NFC reader interaction in the browser or some kind of local-bridge, and communicate with the backend to search, create, update stock records.
- PostgreSQL database : It's the main database for InvenTree and the plugin.


After long reading and understanding of how the BarcodeMixin works, I think we will be going to use ActionMixin rather than BarcodeMixin since this last one use keyboard to write the barcode or just scan for it as an image, there nothing that reassembles to NFC reader like reading a tag directly !! There is always a user intervention either by writing something or searching for it, our NFC solution must be something simple where the user only place the tag and approach it from the reader and then we handle the interaction and prompt him for multiple actions. The barcode approach is kind of slow for NFC.
